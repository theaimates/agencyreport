compiling reports nobody asked for.
Marketing agencies manage multiple clients across multiple platforms. Every week, someone manually opens GA4, Meta Business Suite, Google Ads, Stripe, and Mailchimp — copies numbers into a spreadsheet — writes a summary — formats it — and sends it. Then does it again next week.
📊
Data scattered across 5+ platforms
GA4, Meta Ads, Google Ads, Stripe, and Mailchimp each have separate dashboards, different date pickers, and inconsistent metric definitions. Compiling them requires logging into each one individually.
🕐
4+ hours of manual reporting per client
A single weekly client report — pulling data, writing commentary, formatting the PDF — takes 60–90 minutes per client. A 5-client agency spends half a workday on reports that are out of date by the time they're sent.
📋
No consistency across reports

Zero AI-generated insight
Manual reports present numbers — they rarely explain what the numbers mean or what to do next. The analysis layer gets skipped because there isn't enough time to write it well.

03 — Technical Deep Dive
ReportFlow · Case Study
Six workflows. Each with one job.
Workflow responsibility is strictly scoped — no workflow reads what another writes in the same run. This makes each one independently testable, and means a failure in WF2 never blocks WF6.
Workflow Schedule What it does Type
WF1 — Data Collection Daily 6:00 AM Calls all 5 platform APIs, writes raw rows to Campaigns, Revenue, KPI Snapshots in Airtable Collection
WF2 — Weekly Aggregation Monday 7:00 AM Reads last 7 days of Campaign rows, writes aggregated Channel Performance record per client Aggregation
WF3 — Monthly Aggregation 1st of month 8:00 AM Same as WF2 but for the trailing 30 days — populates monthly Channel Performance records Aggregation
WF4 — AI Report Generation Monday 8:00 AM Reads latest KPI + Channel Performance rows per client, sends structured prompt to GPT-4o, saves HTML body + 3 insights to AI Summaries table AI
WF5 — Email Delivery Monday 9:00 AM Reads all Draft AI Summaries, sends styled HTML email to client contact via Gmail OAuth2, updates status to Sent Delivery
WF6 — Dashboard API On-demand (webhook) Receives client_id from frontend, reads all 6 Airtable tables, returns clean JSON for KPIs, revenue trend, channel mix, campaigns, AI summary API
Key Design Decisions
📦
Airtable as the data warehouse
All platform data lands in Airtable — not a custom database. Free plan works. Every row is human-readable and directly editable. If a collection run fails, you can see exactly which rows are missing and backfill manually.
🔒
No API keys in the frontend
The browser only knows the n8n webhook URL — not a secret. All credentials (Airtable, Meta, Stripe, OpenAI) live inside n8n's encrypted credential store and never touch the client-side code.
🕐
Writers vs Reader — strict separation
WF1–5 only write. WF6 only reads. This means the dashboard never competes with a data collection run. WF6 can be activated and tested with zero data — it just returns empty arrays until WF1 has run.
🤖
Structured prompt engineering for GPT-4o
WF4 sends a strictly formatted JSON payload — not raw text. GPT-4o returns structured HTML with specific insight types (win / watch / next move), making the email template fully predictable and machine-readable by WF5.

Key Takeaways
01
Writers vs Reader is the most important architectural decision
Separating data collection (WF1–5) from data serving (WF6) means the dashboard is always available regardless of whether a scheduled workflow is running. It also means each workflow can be debugged, rerun, or replaced in isolation — without affecting any other part of the system.
02
Airtable outperforms a custom database for small-scale automation
A custom database adds DevOps overhead — hosting, backups, schema migrations. Airtable gives you a queryable, human-readable data warehouse for free. When a workflow misbehaves, you can inspect every row directly in the browser. That debuggability pays for itself constantly.
03
Structured prompts make AI output deterministic enough to automate
WF4 sends GPT-4o a strict JSON schema and asks for output in a defined format (insight_type, title, body_html). This makes WF5 completely predictable — it never needs to parse freeform text. Structured input and output contracts are what separate an AI demo from a production system.
04
No-server frontends compound the value of automation backends
The dashboard is a single HTML file — no Next.js server, no deployment pipeline, no maintenance. It runs in any browser, falls back to mock data when offline, and can be handed to a client who opens it by double-clicking. The automation handles the hard part; the frontend just needs to get out of the way.
